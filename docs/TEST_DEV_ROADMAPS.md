# Roadmaps: перенос на test/dev

## Что меняется

Дорожные карты перестают храниться только в браузере. API и PostgreSQL становятся источником данных для текущего пользователя, поэтому один набор карт открывается одинаково во всех браузерах. Демо-карты не добавляются и не импортируются.

## Связь с обычными задачами

Обычная задача остаётся авторитетным источником заголовка, срока, ответственного и колонки. В существующем payload дорожной карты новыми полями связи являются только `linkedTaskId` и `linkedTaskSnapshot`; отдельная таблица или API связи не нужны. Одна обычная задача может быть связана только с одной полосой во всём доступном наборе карт.

При изменении связанной полосы клиент сначала записывает срок или колонку через task API и только после успешного ответа записывает roadmap API. Соответствие колонки и состояния полосы: `Беклог` → `planned`/0%, `В работе` → `progress`/50%, `Готов` или `Архив` → `done`/100%. Обратное преобразование применяет строгий приоритет: `done` или прогресс ≥100% → `Готов`; иначе `progress` или прогресс >0% → `В работе`; иначе → `Беклог`.

Отвязка и удаление полосы никогда не удаляют обычную задачу. Если задача уже отсутствует, полоса становится независимой. Из `linkedTaskSnapshot` переносятся только валидные строки title/due, поддерживаемая column и валидный owner; пустые, частичные или повреждённые поля не затирают уже валидные значения полосы. Независимые полосы продолжают использовать прежний roadmap payload и прежний API.

Все успешные create/patch/delete обычных задач из доски и архива сразу обновляют общий task cache в `App`; неудачные API-операции cache не меняют. Поэтому при переходе в Roadmaps новые, изменённые и удалённые задачи видны без reload.

## Развёртывание

1. Сделайте архив каталога приложения и резервную копию базы.
2. Установите файлы, затем пересоберите `dashboard-api`: `docker compose -f server/docker-compose.yml up -d --build dashboard-api`.
3. В `frontend` выполните `npm ci && npm run build`; артефакты попадут в `frontend-dist`.
4. Проверьте здоровье: `curl -kfsS https://<host>/api/health`.

## Перенос старых карт

При первом открытии раздела приложение читает прежний ключ `dashboard_roadmaps_v1` только для переноса. Карты с известными demo ID исключаются. Импорт использует `ON CONFLICT DO NOTHING`, поэтому повторный запуск не дублирует данные и не перезаписывает серверные изменения.

## Smoke-проверка в браузере

Локальная цель — `http://localhost:8080`. Из корня репозитория пересоберите frontend и поднимите документированный стек:

```bash
cd frontend
npm ci
npm run build
cd ../server
docker compose up -d --build
curl -fsS http://localhost:8080/api/health
```

Войдите тестовой учётной записью, откройте Roadmaps и последовательно проверьте:

1. Свяжите несвязанную обычную задачу; заголовок, статус, срок и ответственный должны совпасть с Tasks.
2. Измените срок и статус в Roadmaps; в Tasks должны измениться срок и колонка.
3. Перетащите правую границу полосы; срок обычной задачи должен стать новой датой окончания.
4. Отвяжите полосу; обычная задача должна остаться, а полоса — стать независимой.
5. Снова свяжите задачу, затем удалите полосу; обычная задача должна остаться.
6. Обновите страницу; сохранённые связи и изменения должны сохраниться.
7. Создайте, измените и перетащите независимую полосу, затем экспортируйте карту.
8. Повторите обновление в другом браузере под той же учётной записью и убедитесь, что серверное состояние совпадает.
9. На всём проходе в консоли браузера не должно быть ошибок.

### Проверка динамических строк timeline и PDF

1. Откройте в dev дорожную карту `Первые 100 дней ЦАЗС`.
2. В дорожке `Структура` убедитесь, что длинные названия задач переносятся целиком, а каждая полоса выровнена по центру своей строки.
3. Измените ширину окна между узким и широким desktop-размером; подписи и полосы должны оставаться в одной строке.
4. Проверьте линии зависимостей между строками разной высоты: оба конца должны попадать в вертикальные центры соответствующих полос.
5. Перетащите одну полосу и измените её размер, затем отмените или верните тестовые даты, не сохраняя тестовые изменения.
6. Откройте PDF print preview и снова проверьте совпадение подписей, полос и концов линий зависимостей.
7. Убедитесь, что во время всего прохода в консоли браузера нет ошибок.

### Проверка видимости линий зависимостей

1. Для зависимости между задачами с разнесёнными датами проверьте, что линия начинается и заканчивается точно в центрах белых коннекторов, проходит поверх полос, а коннекторы остаются поверх её концов.
2. Для зависимости, где целевая задача начинается сразу после предшественника, проверьте наличие видимого горизонтального плеча не менее 16 px; SVG не должен перехватывать hover, click, drag или resize полос.

## Регрессионная проверка

Перед переносом выполните из корня репозитория:

```bash
cd frontend
node --test $(find src -name '*.test.js' -print | sort)
npm run lint
npm run verify:xlsx
npm run build
```

В запись проверки внесите точное число тестов, число ошибок/предупреждений lint, результат проверки XLSX и build, а также результат каждого шага browser smoke. Если вход или локальный runtime недоступен, пометьте только browser smoke как `BLOCKED` с фактической ошибкой; результаты автоматических проверок не заменяют ручной smoke.
